# CEO 추가 결정 분석 — AD-01~AD-05

**작성일**: 2026-07-21  
**작성자**: 디자인·기술팀  
**상태**: CEO 승인 검토 대기  
**목표**: M3 개발 정책 최종 확정

## 개요

Expert Onboarding의 CEO 12개 기본 결정(Decision 1~12)에 이어, 설계 과정에서 발생한 5개 추가 결정 사항입니다.

| 결정 | 항목 | 특징 | 영향 |
|------|------|------|------|
| 1-9 | 제품 정책 | ✅ APPROVED | M3 개발 필수 |
| 10-12 | 운영 정책 | ✅ APPROVED | M3 운영 필수 |
| AD-01~05 | 추가 정책 | ⏳ PENDING | M3-A 또는 M3-B |

## AD-01: 자격번호 필수 여부

### 배경

자격·면허 입력 화면(EXP-ONB-005)에서 다음을 선택:
- **A안**: 자격번호 필수 (모든 자격 등록 시 필수)
- **B안**: 자격번호 선택 (기관명·취득일만으로 충분)

### 분석

#### A안: 자격번호 필수

**장점**:
- ✅ 자격 검증 가능성 증대 (번호 기반 검증)
- ✅ 중복 등록 방지 (같은 번호 중복 체크)
- ✅ 법적 관련 기록 명확 (번호 추적 가능)

**단점**:
- ❌ 사용자 진입장벽 증가 (기억해야 할 정보 많음)
- ❌ 번호 모르는 자격 입력 불가 (완료도 저하)
- ❌ 국제 자격 미대응 (번호 없는 자격들)

**구현 영향**:
- 필수 필드: profiles.license_number (또는 licenses.license_number)
- 검증: 정규식 또는 외부 API
- 중복 체크: DB 제약 또는 비즈니스 로직

#### B안: 자격번호 선택

**장점**:
- ✅ 낮은 진입장벽 (필수 정보 최소화)
- ✅ 다양한 자격 대응 (번호 없는 자격도 등록)
- ✅ 초기 완료도 높음 (사용자 만족도)

**단점**:
- ❌ 자격 검증 어려움 (허위 입력 확인 곤란)
- ❌ 검토팀 수작업 증가 (관리자 검증)
- ❌ 법적 책임 불명확 (증빙 부족)

**구현 영향**:
- 선택 필드: licenses.license_number
- 검증: 없음 (UI 레벨 선택사항)
- 중복 체크: 불필요

### 권장안

**B안 (자격번호 선택)** 추천

**근거**:
1. **초기 서비스 우선 완료도**: Expert Onboarding은 초기 프로필 작성이 목표이므로, 완료 장벽을 낮추는 것이 우선
2. **검토 프로세스 신뢰**: 자격·면허 검증은 관리팀의 검토 프로세스에서 담당 가능 (번호 요청·검증)
3. **증빙파일 활용**: 자격번호보다 증빙파일(자격증 이미지)이 더 강력한 검증 수단
4. **향후 개선 가능**: 서비스 확대 시 필수화 가능

**단, 다음 조건**:
- 검토 승인 시점에 번호 필수 요청 가능 (수동)
- 증빙파일 필수 (AD-02, AD-03와 함께)

### Screen Spec 반영

```markdown
**선택 입력**:
- [ ] 자격번호
  - 선택사항 (제출 시 필수 아님)
  - 형식: 각 기관별 상이 (미검증)
  - 검토팀: 승인 시 번호 확인 가능
```

### M3 입력

**M3-B 단계**:
- licenses 테이블: license_number 필드 선택 추가
- 검증 로직: 없음 (선택)

## AD-02: 증빙파일 허용 형식

### 배경

자격 증빙 업로드 화면(EXP-ONB-006)에서:
- 어떤 파일 형식을 허용할 것인가?

### 분석

#### 후보 형식

**현실적 옵션**:
1. **PDF만** 
   - 장점: 표준화, 위변조 어려움
   - 단점: 모바일 캡처 불편

2. **PDF + 이미지 (JPG/PNG)**
   - 장점: 모바일 친화적 (카메라 캡처)
   - 단점: 파일 크기 커짐

3. **제한 없음 (일반 파일)**
   - 장점: 사용자 편의 극대화
   - 단점: 악의적 파일 업로드 위험

#### 규모 추정

| 형식 | 평균 크기 | 1000명 기준 |
|------|---------|----------|
| JPG (자격증 1장) | 500KB | 500MB |
| PDF | 800KB | 800MB |

**S3 비용**: 월 ~$10-20 (저용량)

### 권장안

**PDF + JPG/PNG** 허용

**근거**:
1. **균형**: 모바일 사용성 + 파일 표준화
2. **보안**: 이미지·PDF만 필터 (실행파일 차단)
3. **관리**: 파일 형식 검증 간단

**단, 용량 제한** (AD-03):
- 단일 파일: 5MB 이하
- 자격당 최대 1개 파일

### Screen Spec 반영

```markdown
**필수 입력**:
- [ ] 등록한 모든 자격의 증빙파일
  - 지원 형식: PDF, JPG, PNG
  - 최대 용량: 5MB (파일당)
```

### M3 입력

**M3-B 단계**:
- Storage 정책: .pdf, .jpg, .png만 허용
- 파일명: uuid.{ext} 형식 (변경 불가)

## AD-03: 증빙파일 최대 용량

### 배경

증빙파일 용량 제한:
- 너무 작으면: 사진 품질 낮음 (검토 불가)
- 너무 크면: S3 비용 + 업로드 시간

### 분석

#### 용량별 평가

| 용량 | 이미지 품질 | S3 비용/년 | 업로드 시간 | 평가 |
|------|----------|---------|----------|------|
| 2MB | 낮음 (부분 잘림) | ~$10 | <5초 | ❌ 불충분 |
| 5MB | 양호 (전체 선명) | ~$20 | 5-10초 | ✅ 권장 |
| 10MB | 매우 좋음 (고해상도) | ~$40 | 10-20초 | ⚠️ 과다 |

### 권장안

**5MB 제한** (AD-02와 함께)

**근거**:
1. **자격증 사진**: A4 스캔본 또는 휴대폰 사진 기준
2. **비용 효율**: S3 월 ~$15-20 (저비용)
3. **사용성**: 대부분의 사진 형식 수용

### Screen Spec 반영

```markdown
- [ ] 최대 용량: 5MB (파일당)
  - 초과 시: 파일 선택 거부 (업로드 불가)
```

### M3 입력

**M3-B 단계**:
- Storage 정책: multipart upload 최대 5MB
- 클라이언트: 업로드 전 파일 크기 검증

## AD-04: 공개범위 (센터명·주소·홈페이지)

### 배경

근무기관 정보의 공개 여부:
- **센터명**: 기관을 특정하는 정보
- **주소**: 위치 정보 (거주지역과 다름)
- **홈페이지**: 기관 대표 정보

현재 상태: **소유자 전용** 또는 **선택적 공개** 미결정

### 분석

#### A안: 소유자 전용 (공개 금지)

**가정**: 모든 근무기관 정보 비공개

**장점**:
- ✅ 개인정보 보호 (직무 정보 노출 최소)
- ✅ 프라이버시 우수 (고평가 기대)
- ✅ 구현 간단 (RLS: user_id = auth.uid)

**단점**:
- ❌ 프로필 신뢰도 저하 (근무처 알 수 없음)
- ❌ 검색·필터링 불가능 (지역별 전문가 찾기 안 됨)
- ❌ 플랫폼 가치 제한 (매칭 기능 제약)

#### B안: 선택적 공개

**가정**: 사용자가 선택하도록

**세부 옵션**:
- B-1: 센터명만 공개 (주소·홈페이지는 비공개)
- B-2: 센터명 + 홈페이지 (주소는 비공개)
- B-3: 모두 공개 (사용자 선택)

**B-1 평가**:
- ✅ 균형: 기관 정보 + 위치 정보 분리
- ✅ 신뢰도: 센터명 확인 가능
- ✅ 개인정보: 정확한 주소 숨김

**B-2 평가**:
- ✅ 기관 검증: 홈페이지 통해 확인 가능
- ⚠️ 위치 정보: 여전히 숨김

**B-3 평가**:
- ✅ 투명성 극대화
- ❌ 개인정보 노출 (위치 추적 가능)

#### C안: 승인 후 공개

**가정**: 관리팀이 검토 후 공개 여부 결정

**장점**:
- ✅ 최고 수준 보호 (검토팀 판단)
- ✅ 위험 사전 차단

**단점**:
- ❌ 구현 복잡 (승인 상태별 공개 로직)
- ❌ 사용자 선택권 없음 (주도권 상실)

### 권장안

**B-1: 센터명 + 홈페이지 / 주소는 비공개** (또는 **사용자 선택**)

**근거**:
1. **균형**: 근무처 신뢰도 + 위치 개인정보 보호
2. **실용성**: 센터명으로 대부분의 신뢰 확보 가능
3. **선택지**: 사용자가 공개 여부 선택 가능하면 더 좋음

**대안 (더 보수적)**:
- 소유자 전용 (A안) → 향후 선택적 공개로 개선

### Screen Spec 반영

```markdown
**공개 범위** (AD-04):

- [ ] 센터명: 선택적 공개
  - 공개 선택: 전체 노출
  - 비공개 선택: 검토 승인 시에만 관리팀 확인
  
- [ ] 홈페이지: 선택적 공개 (센터명 공개 시)
  
- [ ] 주소: 항상 비공개 (관리팀만 조회 가능)
  - 예외: 거주지역은 별도 정책 (TM-06)
```

### M3 입력

**M3-A 단계**:
- workplaces.is_center_public (BOOLEAN)
- RLS: is_center_public = true인 경우만 조회 가능

**M3-B 단계**:
- 관리자 대시보드: 비공개 기관정보 조회 (admin role)

## AD-05: 지역 입력 단위와 옵션 체계

### 배경

거주지역·근무지역 입력:
- 현재: 드롭다운 (마스터 데이터)
- 문제: 어떤 단위로 제공할 것인가?

### 분석

#### 옵션 1: 광역시도 단위 (17개)

```
서울, 부산, 대구, 인천, 광주, 대전, 울산
세종, 경기, 강원, 충북, 충남, 전북, 전남, 경북, 경남, 제주
```

**장점**:
- ✅ 옵션 적음 (17개 드롭다운 간단)
- ✅ 모두가 이해 (광역단위 표준)

**단점**:
- ❌ 정보 부족 (시군구 구분 없음)
- ❌ 필터링 정확도 낮음 (예: 서울 강남과 강북 같음)

#### 옵션 2: 시군구 단위 (243개)

```
서울 (25개): 강남, 강북, 강동, ...
부산 (16개): 강서, 금정, 남구, ...
...
```

**장점**:
- ✅ 정확한 위치 정보 (지역 특성 반영)
- ✅ 필터링 정확도 높음

**단점**:
- ❌ 옵션 많음 (243개 드롭다운, 최악)
- ❌ 사용자 입력 부담 (2단계 선택 필요)

#### 옵션 3: 광역시도 + 주요 시급 (40개)

```
서울, 부산, 대구, 인천, ...
서울 내: 강남, 강북, 서초, ...
부산 내: 해운대, 사상, 사하, ...
```

**장점**:
- ✅ 균형 (중요 지역 구분)
- ✅ 사용성 (2단계 선택, 드롭다운 범위 관리)

**단점**:
- ⚠️ 분류 기준 모호 (어디까지가 "주요"인가)

### 권장안

**옵션 3: 2단계 선택 (광역 + 시급)**

**구현**:

```
1단계: 광역시도 선택 (17개)
  ↓
2단계: 해당 시도의 시급 선택 (5~30개)
```

**근거**:
1. **사용성**: 드롭다운 관리 가능한 수준
2. **정보 완성도**: 시군구 수준의 정보 제공
3. **검색·필터링**: 지역 특성 반영 가능

**마스터 데이터 예시**:
```
서울: 강남구, 강북구, 강동구, 관악구, ...
부산: 강서구, 금정구, 남구, 동구, ...
```

### Screen Spec 반영

```markdown
**선택 입력**:
- [ ] 거주지역 (2단계 선택)
  1단계: 광역시도
  2단계: 시군구 (선택)
  
- [ ] 근무지역 (2단계 선택)
  정책: 거주지역과 독립적
```

### M3 입력

**M3-A 단계**:
- regions 테이블: id, parent_id (NULL이면 광역), name, level (1 or 2)
- workplaces: residence_region_id, workplace_region_id (FK regions)

**마스터 데이터**: 별도 migration (2026-07-22 내로)
