## 3. CEO 최종 결정사항

### 3.1 이름 또는 활동명

```
최소 2자
최대 40자
```

타인 사칭, 자격 오인 또는 부적절한 표현은 관리자 검토 대상이다.

---

### 3.2 프로필 사진 정책

```
프로필 작성 중:
선택사항 (사진 없이 저장 가능)

검토 요청 시:
필수 (없으면 제출 차단)

승인된 공개 프로필:
필수
```

사용자는 사진 없이 작성 내용을 임시 저장할 수 있다.

검토 요청 단계에서는 사진이 없으면 제출을 차단한다.

---

### 3.3 프로필 사진 형식

지원 형식은 다음과 같다.

```
JPEG
PNG
WebP
```

실제 확장자와 MIME type을 함께 검증한다.

HEIC 등 추가 형식은 MVP 범위에 포함하지 않는다.

---

### 3.4 프로필 사진 용량

```
최대 업로드 용량:
5MB
```

압축, 리사이징 및 최종 저장 규칙은 M2 기술 검증에서 권고안을 작성한다.

---

### 3.5 직군 목록

MVP 직군은 다음 8개로 운영한다.

```
1. 물리치료사
2. 퍼스널트레이너
3. 건강운동관리사
4. 선수트레이너
5. 필라테스 지도자
6. 요가 지도자
7. 스포츠 코치
8. 기타 운동·재활 전문가
```

**원칙**:
- 직군 선택은 면허나 자격 인증을 의미하지 않는다.
- 면허와 자격은 자격·면허 정보 및 증빙자료를 통해 별도로 검증한다.
- 직군 목록은 향후 서비스 운영 결과와 CEO 승인에 따라 수정할 수 있다.

---

### 3.6 한 줄 소개

한 줄 소개는 **선택사항**이다.

```
필수 여부:
선택

최소 글자 수:
없음

작성 시 최대 글자 수:
60자
```

한 줄 소개가 없다는 이유로 다음 단계 이동이나 검토 요청을 차단하지 않는다.

---

### 3.7 상세 소개

상세 소개는 **선택사항**이다.

```
필수 여부:
선택

최소 글자 수:
없음

작성 시 최대 글자 수:
500자
```

상세 소개를 작성하지 않았다는 이유로 검토 요청을 차단하지 않는다.

**Future Extension**:
- JPEG, PNG, WebP 등 이미지 파일을 활용한 상세 소개 자료 등록 가능성 검토
- MVP에 포함하지 않음: 이미지 UI, 전용 Storage, DB 필드, 다중 이미지 관리, PDF 포트폴리오

---

### 3.8 거주지역·근무지역 정책

거주지역과 근무지역을 별도 정보로 구분한다.

#### 거주지역

```
입력:
전문가가 설정 가능

공개 프로필:
노출하지 않음

다른 사용자·소비자:
조회 불가

전문가 본인:
조회·수정 가능
```

- 거주지역은 정확한 주소가 아니라 지역 단위 정보로 수집한다.
- 거주지역의 구체적인 입력 단위는 M2 검토 후 확정한다.

#### 근무지역

```
입력:
전문가가 설정 가능

공개 여부:
전문가가 선택

공개 선택 시:
공개 프로필 및 검색 결과에 표시 가능

비공개 선택 시:
소비자와 다른 사용자에게 노출하지 않음
```

- 근무지역과 근무기관의 상세 주소는 서로 다른 정보로 관리한다.
- 근무기관 주소와 근무지역의 저장 구조 관계는 M2에서 확인한다.

---

### 3.9 지역정보 공개 범위

공개 범위 분류를 다음과 같이 통합한다.

| 분류 | 데이터 | 접근 권한 | 노출 범위 |
|------|--------|---------|---------|
| **공개 프로필** | 승인된 기본정보, 경력, 공개 자격정보, 교육, 전문분야, 공식 연락처, 근무지역(선택 시) | 누구나 | 소비자 공개 프로필 표시 |
| **소유자 제한** | 개인 연락처, 자격번호, 증빙파일 메타데이터, **거주지역** | 해당 전문가 + 권한 관리자 | 공개 프로필 미노출 |
| **선택적 공개** | **근무지역** | 해당 전문가 + 권한 관리자 | 전문가 공개 선택 시만 표시 |
| **당사자 제한** | 사용자 전달용 반려 사유 (user_rejection_reason) | 해당 전문가 + 권한 관리자 | 관리자 기록 미노출 |
| **관리자 전용** | 내부 관리자 메모 (internal_admin_notes) | 권한 있는 관리자만 | 어떤 사용자에게도 노출 금지 |

**원칙**:
- 거주지역은 공개 프로필, 전문가 검색 결과 또는 다른 사용자 화면에 표시하지 않는다.
- 근무지역은 전문가가 공개를 선택한 경우에만 표시한다.

---

### 3.10 주소 입력 방식

MVP에서는 수동 입력을 기본으로 한다.

```
주소 수동 입력:
포함 (필수)

주소 검색:
MVP 제외

주소 자동완성:
MVP 제외

지도 SDK:
MVP 제외

지도 마커 선택:
MVP 제외
```

**Post-MVP 검토**:
- 주소 검색, 자동완성, 정규화
- 지역 자동 분류
- 외부 주소 서비스 연동

**원칙**:
- 특정 서비스 제공업체를 현재 문서에서 확정하지 않는다.
- 외부 API 또는 MCP 연동 여부는 Post-MVP 기술 검토 후 CEO 승인을 받아 결정한다.

---

### 3.11 검토 요청 후 수정 정책

프로필 상태별 수정 가능 여부는 다음과 같다.

```
Draft:
수정 가능

Pending (검토 중):
수정 불가

Rejected (반려됨):
수정 가능 → 재제출 가능

Approved (승인됨):
별도 정책으로 관리
```

**원칙**:
- 검토 중인 데이터가 임의로 변경되지 않도록 Pending 상태에서는 편집 기능을 차단한다.
- Approved 상태에서의 수정 및 재검토 정책은 별도로 수립한다.

---

### 3.12 예상 검토 기간 표시

사용자에게 예상 검토 기간을 다음과 같이 안내한다.

```
예상 검토 기간:
1~3일
```

**권장 안내문구**:

```
프로필 검토에는 일반적으로 1~3일이 소요됩니다.
검토 결과는 마이페이지에서 확인할 수 있습니다.
```

**주의**:
- 이는 법적 또는 절대적인 처리기한 보장이 아니라 일반적인 예상 기간이다.
- 운영 상황에 따라 실제 검토가 지연될 수 있다는 안내를 함께 표시할 수 있다.
- 이메일 알림 기능이 확정되지 않은 상태에서는 이메일 안내를 전제로 한 문구를 사용하지 않는다.

## 7. M2 필수 검증 근거

각 항목에는 가능한 범위에서 다음 근거를 첨부한다.

```
• 관련 파일명과 경로
• 현행 Migration 파일
• Migration 버전 및 작성일
• 관련 Git commit
• 현재 DB 적용 여부
• 현재 RLS 정책
• 현재 Storage 정책
• 관련 코드 위치
• Screen Spec과 실제 구현의 차이
• 기술 권고사항
```

**상태 표기**:

```
Verified:
근거 확인 완료 (Migration + Commit + DB 적용)

Pending Mapping:
구조 확인 필요 (M2 검증 대기)

New Field / Technical Proposal:
새로운 구조 필요
```

근거가 충분하지 않은 항목은 `Verified`로 표시하지 않는다.

## 8. M2 작업 범위

### M2에서 허용되는 작업

✅ 현재 코드와 DB 구조 확인
✅ 기존 Migration 파일 확인
✅ 현재 RLS 정책 분석
✅ 현재 Storage Bucket과 접근 정책 검토
✅ Screen Spec과 실제 시스템의 차이 기록
✅ 필요한 필드와 권한 구조 제안
✅ 기술 리스크 분석
✅ 구현 권고안 작성
✅ 예상 변경 범위 정리
✅ CEO 결정사항의 기술적 실현 가능성 검토

### 별도 승인 없이 진행하지 않는 작업

❌ 신규 Migration 작성 및 적용
❌ Production DB 변경
❌ RLS 정책 변경
❌ Storage 정책 변경
❌ 신규 Bucket 생성
❌ 신규 DB 필드 생성
❌ 외부 주소 서비스 연동
❌ 신규 테이블 생성
❌ Production 배포

**원칙**:

M2 기술 검증 일정이 승인됐다는 사실이 실제 DB 변경까지 자동으로 승인한다는 뜻은 아니다.

기술 변경이 필요한 경우 다음 절차를 따른다.

```
M2 현행 구조 검증
   ↓
변경 필요사항 및 권고안 작성
   ↓
CTO 검토 및 승인
   ↓
CEO 보고 및 최종 승인
   ↓
승인된 기술 변경만 실행
```

## 10. 디자인팀 작업 지시

## 15. 주의사항

### 거주지역·근무지역 분리

이것은 기존 "지역" 필드의 큰 변화다.

```
기존: 지역 (1개 필드)
현재: 거주지역 (비공개) + 근무지역 (선택적 공개)
```

- M2에서 저장 구조를 재검토해야 한다.
- 거주지역 RLS는 "해당 전문가만 조회·수정" 구현 필요
- 근무지역은 공개 여부 flag 필요

### 검토 후 수정 정책

Rejected 상태에서만 수정 가능하다.

```
Pending (검토 중): 수정 불가
Rejected (반려됨): 수정 가능 → 재제출
```

- 편집 UI 권한 관리 필요
- 상태 전환 로직 명확화 필요

### 검토 기간 표시

"1~3일"은 예상 기간이며 보장이 아니다.

```
✓ 안내 가능: "일반적으로 1~3일 소요"
✗ 금지: "반드시 1~3일 내 처리"
✗ 금지: "이메일로 완료 알려드립니다" (이메일 미확정)
```
